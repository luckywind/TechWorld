以前写代码占工作时间的 70%，现在大概只占 20%。剩下的 80% 在定义问题、审阅结果、维护结构。这个比例的变化，才是 Vibe Coding 在说的事。

AI 是一个忠实的放大器。你走对了路，它帮你加速。你走错了路，它也帮你加速，速度一样快，甚至更快，因为走错路的时候你不会停下来仔细想。

✅写好代码的几个最佳实践：

1. 原子化。把不同性质的问题拆到足够小的单元里。以贪吃蛇为例：状态推进放在纯逻辑层，渲染层只管把状态画出来，输入层只管收集事件并翻译成统一信号，持久化放到边界模块。规则改了不用动渲染，界面改了不用碰核心逻辑，换一种输入方式也不用重写业务代码。

2. 接口先于实现。很多人还没想清楚模块的输入、输出、错误语义和边界条件，就急着让 AI 给一个”完整方案”。拿到的东西看着像那么回事，工程上经不起推敲。更靠谱的做法是先把接口契约定好，接收什么、返回什么、什么情况下失败、失败时暴露什么信息，再让 AI 在这个框架内生成实现。这条在 Vibe Coding 场景下最容易被忽略，但也最管用。
3. 机制和策略要分开。
4. 文本化。需求草稿、结构化配置、模块变更的说明文字，都应该以可读、可审、可版本化的形式留存

## 关于提示词

Prompt 不该是一次性的聊天记录，而该当作可审阅、可迭代、可复用的设计文档来管理。我在团队里推过这个做法，一开始大家觉得麻烦，后来发现返工次数少了不少。

一套实用的 Prompt 结构可以分成六段：Context 定义上下文，Goal 声明目标，Constraints 列出边界和约束，Interface & Data 约定接口数据，Process 规定执行路径要求 AI 分步产出，Review 明确验收标准。不一定要死守这个格式，但覆盖到这几个要素，生成结果会稳定不少。

还有一个技巧：需求含糊的时候，别直接要代码，先让 AI 提出问题。比如：”先不要写代码。请基于以下业务描述提出 8 个澄清问题，覆盖目标用户、核心流程、失败场景、性能约束、兼容范围和上线节奏。等回答后，再输出实现方案。”看着慢，但经常能省掉后续数小时的返工。
